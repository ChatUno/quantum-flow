## ⚡ STEP 2: FINITE_STATE_MACHINE (FSM) ONTOLOGY

**GlobalStateObject Schema:**

```javascript
const GlobalState = {
  currentView: 'gateway', // gateway | archive | stress | latency
  requests: [], // Array of intercepted webhooks
  filteredRequests: [], // Filtered view
  stressTest: {
    active: false,
    url: '',
    count: 0,
    success: 0,
    error: 0,
    history: []
  },
  latencyData: [], // Array of {timestamp, latency}
  filter: {
    query: '',
    statusCode: null,
    headers: {}
  },
  selectedRequest: null, // Currently viewed request
  status: 'IDLE' // IDLE | PROCESSING | SUCCESS | ERROR | EMPTY
}
```

**State Enumeration:**
- **IDLE:** System ready, waiting for input
- **PROCESSING:** Request in flight, stress test active
- **SUCCESS:** Operation completed successfully
- **ERROR:** Operation failed, recovery needed
- **EMPTY:** No data available (archive empty, no requests)

**State Transitions:**

```
IDLE → PROCESSING: User sends request / starts stress test
PROCESSING → SUCCESS: Request completes / stress test finishes
PROCESSING → ERROR: Request fails / timeout
ERROR → IDLE: User acknowledges error
SUCCESS → IDLE: Auto-transition after display
EMPTY → IDLE: User loads data / receives first request
IDLE → EMPTY: User clears archive / no data
```

**Critical Validation:**
- ✓ Every terminal state has transition back to IDLE
- ✓ Every state reachable from IDLE
- ✓ No dead-end states
- ✓ Complete session lifecycle supported

## ⚡ STEP 2.5: RESILIENCE_PATTERNS

**Pattern 1: Graceful Degradation (IndexedDB Failure)**
- **Scenario:** Browser blocks IndexedDB
- **Strategy:** Fallback to in-memory array with warning
- **Stress Test:** Disable IndexedDB in browser settings

**Pattern 2: State Recovery (Session Loss)**
- **Scenario:** User refreshes page
- **Strategy:** Rehydrate from IndexedDB on load
- **Stress Test:** Hard refresh with 1000+ archived requests

**Pattern 3: Memory Self-Regulation**
- **Scenario:** Archive grows beyond browser limits
- **Strategy:** Auto-prune oldest records beyond 10,000
- **Stress Test:** Inject 50,000 requests, verify auto-cleanup

**Pattern 4: Adaptive Performance**
- **Scenario:** Low-end device, 60fps graph impossible
- **Strategy:** Reduce graph update rate to 30fps dynamically
- **Stress Test:** Simulate CPU throttling in DevTools

- No cloud transmission
- User controls data deletion
- Export includes all data for backup

- URL validation before fetch
- JSON parsing with try/catch
- Header sanitization
- Status code validation

**Threat Model:**

- **Injection:** Prevented via JSON parsing
